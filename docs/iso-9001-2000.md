# Implementasi ISO 9001:2000 pada Sistem SnackCheck

## 1. Gambaran Umum

SnackCheck merupakan sistem pengelolaan hasil pengujian kualitas produk makanan. Sistem mendukung proses pengiriman hasil uji, review oleh QC Manager, pemberian keputusan akhir, penerbitan Certificate of Analysis, pengiriman notifikasi, dan pencatatan audit trail.

Implementasi ISO 9001:2000 pada SnackCheck bertujuan memastikan proses pengembangan dan operasional sistem berjalan secara terdokumentasi, terukur, terkendali, dan terus diperbaiki.

Parameter yang diterapkan meliputi:

1. Dokumentasi proses.
2. Pengendalian kualitas.
3. Continuous improvement.
4. Corrective Action.
5. Preventive Action.

## 2. Dokumentasi Proses

Dokumentasi proses digunakan agar setiap aktivitas pengembangan dan penggunaan sistem dapat dipahami, diulang, dan diperiksa.

Dokumen yang digunakan pada proyek SnackCheck meliputi:

1. Dokumentasi kebutuhan sistem.
2. User story fitur submit hasil uji.
3. User story detail batch.
4. User story keputusan akhir QC.
5. User story penerbitan CoA.
6. User story notifikasi.
7. Struktur database dan migration.
8. Dokumentasi source code melalui repository GitHub.
9. Test case regression testing.
10. Dokumentasi pengujian Selenium.
11. Dokumentasi security testing.
12. Bukti screenshot hasil pengujian.
13. Riwayat commit dan perubahan kode.

Alur utama proses SnackCheck didokumentasikan sebagai berikut:

1. Analis memasukkan hasil pengujian.
2. Sistem mengubah status batch menjadi menunggu_review.
3. QC Manager membaca detail batch dan parameter pengujian.
4. QC Manager memilih keputusan lulus, tidak_lulus, ditahan, atau uji_ulang.
5. Sistem memvalidasi keputusan.
6. Jika keputusan lulus, sistem membuat dokumen CoA.
7. Sistem mengirimkan notifikasi sesuai keputusan.
8. Sistem mencatat perubahan pada audit trail.

Dokumentasi tersebut membantu tim memahami hubungan antara kebutuhan, implementasi, pengujian, dan hasil akhir sistem.

## 3. Pengendalian Kualitas

Pengendalian kualitas dilakukan untuk memastikan sistem memenuhi kebutuhan fungsional dan tidak menghasilkan keputusan yang salah.

Pengendalian kualitas pada SnackCheck meliputi:

### 3.1 Validasi Input

Sistem melakukan validasi terhadap:

1. ID batch.
2. Keputusan akhir QC.
3. Tindakan rekomendasi.
4. Catatan keputusan.
5. Status batch.
6. Nilai parameter hasil pengujian.

Contohnya, keputusan tidak_lulus harus memiliki tindakan rekomendasi seperti rework, disposal, atau hold.

### 3.2 Regression Testing

Regression testing dilakukan untuk memastikan perubahan sistem tidak merusak fungsi yang sudah berjalan.

Pengujian yang telah diterapkan meliputi:

1. Membuka halaman utama.
2. Melihat detail batch.
3. Menangani batch yang tidak ditemukan.
4. Submit hasil pengujian.
5. Menyimpan keputusan lulus.
6. Membuat CoA.
7. Menolak keputusan tidak_lulus tanpa rekomendasi.
8. Menyimpan keputusan tidak_lulus dengan rework.
9. Menyimpan keputusan uji_ulang tanpa CoA.
10. Menyimpan keputusan ditahan tanpa CoA.
11. Mencatat audit trail.
12. Mencegah duplikasi CoA.

Hasil pengujian menunjukkan 12 test case berhasil dijalankan.

### 3.3 Automated Testing

Selenium dan pytest digunakan untuk menjalankan pengujian melalui browser.

Automated testing membantu:

1. Mengurangi kesalahan pengujian manual.
2. Mempercepat pengujian berulang.
3. Menyediakan bukti screenshot.
4. Memastikan hasil pengujian konsisten.
5. Membantu proses verifikasi sebelum perubahan kode digabungkan.

### 3.4 Audit Trail

Setiap perubahan keputusan dicatat melalui audit trail.

Informasi yang dicatat meliputi:

1. Status lama.
2. Status baru.
3. Waktu perubahan.
4. Jenis tindakan.
5. Data keputusan.

Audit trail mendukung proses pemeriksaan ketika terjadi kesalahan atau insiden.

### 3.5 Kontrol Dokumen CoA

Dokumen CoA hanya dibuat ketika batch dinyatakan lulus.

Pengujian juga memastikan:

1. CoA tidak dibuat untuk status ditahan.
2. CoA tidak dibuat untuk status uji_ulang.
3. CoA tidak dibuat untuk status tidak_lulus.
4. CoA tidak terduplikasi ketika keputusan lulus dikirim lebih dari satu kali.
5. File CoA tersimpan dan dapat diakses.

## 4. Continuous Improvement

Continuous improvement dilakukan dengan mengevaluasi hasil pengujian dan menggunakan temuan tersebut untuk memperbaiki sistem.

Proses continuous improvement pada SnackCheck meliputi:

1. Menjalankan regression testing setelah perubahan.
2. Mencatat test case yang gagal.
3. Menganalisis penyebab kegagalan.
4. Memperbaiki kode atau skenario test.
5. Menjalankan ulang test.
6. Menambahkan test case baru ketika ditemukan bug.
7. Meninjau hasil security testing.
8. Memperbarui dokumentasi.
9. Menyimpan perubahan melalui Git.
10. Mengevaluasi risiko sebelum release.

Contoh penerapan continuous improvement terjadi pada RT-08, RT-09, dan RT-12.

Pada RT-08, test gagal karena data rework dicari pada elemen yang salah. Skenario kemudian diperbaiki dengan membaca data dari response API.

Pada RT-09, test diperbaiki agar menunggu response detail batch selesai dimuat.

Pada RT-12, test diperbaiki untuk menghindari StaleElementReferenceException dengan membaca ulang elemen setelah halaman berubah.

Setelah perbaikan, seluruh 12 regression test berhasil dijalankan.

## 5. Corrective Action

Corrective Action merupakan tindakan untuk memperbaiki masalah yang sudah ditemukan.

Temuan utama pada SnackCheck adalah Broken Access Control.

Security testing menunjukkan bahwa pengguna tanpa login dapat:

1. Membuka halaman sistem.
2. Melihat detail batch.
3. Submit hasil pengujian.
4. Mengubah keputusan akhir menjadi lulus.
5. Memicu penerbitan CoA.

Corrective Action yang perlu dilakukan meliputi:

1. Menambahkan sistem login.
2. Menambahkan middleware auth pada route kritis.
3. Membatasi keputusan akhir hanya untuk QC Manager.
4. Menambahkan Role-Based Access Control.
5. Menghapus bypass authorization yang tidak diperlukan.
6. Menambahkan Laravel Policy atau Gate.
7. Menolak pengguna anonim dengan HTTP 401.
8. Menolak pengguna tanpa role QC Manager dengan HTTP 403.
9. Mencatat identitas pengguna pada audit trail.
10. Menjalankan ulang security testing setelah perbaikan.

Verifikasi Corrective Action dilakukan dengan memastikan pengguna anonim tidak lagi dapat mengakses fungsi kritis.

## 6. Preventive Action

Preventive Action merupakan tindakan untuk mencegah masalah sebelum terjadi.

Tindakan pencegahan pada SnackCheck meliputi:

1. Membuat checklist keamanan sebelum release.
2. Menjalankan regression testing sebelum merge.
3. Menjalankan security testing secara berkala.
4. Menerapkan code review.
5. Menggunakan branch khusus untuk perubahan fitur.
6. Melarang penyimpanan data sensitif pada repository.
7. Membatasi akses berdasarkan role.
8. Membuat backup database.
9. Menambahkan monitoring log.
10. Menambahkan automated authorization testing.
11. Menetapkan quality gate sebelum deployment.
12. Memeriksa migration sebelum diterapkan.
13. Menyimpan bukti pengujian.
14. Menggunakan audit trail untuk aktivitas kritis.
15. Meninjau dependency melalui Composer Audit dan npm audit.

Preventive Action bertujuan mengurangi kemungkinan terjadinya kesalahan keputusan, kehilangan data, akses tidak sah, dan penerbitan CoA yang tidak valid.

## 7. Siklus Penerapan Kualitas

Siklus pengelolaan kualitas pada SnackCheck dapat dijelaskan sebagai berikut:

### Plan

1. Menentukan kebutuhan sistem.
2. Menentukan risiko perubahan.
3. Menyusun test case.
4. Menentukan prioritas pengujian.

### Do

1. Mengembangkan fitur.
2. Menjalankan sistem.
3. Melakukan pengujian manual.
4. Menjalankan automated testing.

### Check

1. Memeriksa hasil regression testing.
2. Memeriksa security testing.
3. Memeriksa log.
4. Memeriksa audit trail.
5. Memeriksa dokumen CoA.

### Act

1. Memperbaiki defect.
2. Menambah test case.
3. Memperbarui dokumentasi.
4. Menambahkan kontrol keamanan.
5. Mencegah masalah serupa.

## 9. Kesimpulan

Penerapan ISO 9001:2000 pada SnackCheck dilakukan melalui dokumentasi proses, pengendalian kualitas, perbaikan berkelanjutan, Corrective Action, dan Preventive Action.

Regression testing menunjukkan seluruh 12 test case berhasil. Namun, security testing menemukan Broken Access Control dengan tingkat risiko Critical.

Temuan tersebut menjadi dasar Corrective Action dan Preventive Action agar sistem memiliki kontrol autentikasi dan otorisasi yang lebih baik sebelum digunakan pada lingkungan produksi.
